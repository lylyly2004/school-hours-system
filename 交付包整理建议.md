# 轻课管教培课时管理系统 交付包整理建议

## 一、建议交付包名称

建议正式交付时，将系统整理成单独文件夹或压缩包，命名方式例如：

- `轻课管教培课时管理系统_V1.0`
- `轻课管教培课时管理系统_本地版_V1.0`

如果以后持续更新，可继续使用：

- `轻课管教培课时管理系统_V1.1`
- `轻课管教培课时管理系统_V1.2`

---

## 二、建议交付目录结构

建议最终交付给客户时，整理成如下结构：

```text
轻课管教培课时管理系统_V1.0
├─ 系统程序
│  ├─ index.html
│  ├─ pages
│  ├─ css
│  ├─ js
│  └─ images
├─ 数据文件
│  └─ school-hours-data.json
├─ 使用说明
│  ├─ 快速上手.txt
│  └─ 使用说明.md
└─ 更新与售后
   ├─ 版本信息.txt
   └─ 联系方式.txt
```

---

## 三、各文件夹建议用途

### 1. 系统程序

放系统运行需要的全部网页文件和资源文件，包括：

- `index.html`
- `pages`
- `css`
- `js`
- `images`

客户正常使用时，只需要打开：

- `系统程序 -> index.html`

### 2. 数据文件

建议单独放一个默认数据文件，例如：

- `school-hours-data.json`

作用：

- 首次绑定数据文件时可以直接选择
- 后续导出备份文件也方便统一放在这里
- 便于换电脑迁移

### 3. 使用说明

建议放两份说明：

- `快速上手.txt`
  - 适合第一次快速查看
- `使用说明.md`
  - 适合完整阅读、长期留存

### 4. 更新与售后

建议单独放这两类文字说明：

- `版本信息.txt`
  - 记录版本号、更新时间、更新内容
- `联系方式.txt`
  - 写清你的联系方式、售后说明

---

## 四、建议交付时一并准备的文件

### 1. 版本信息.txt

建议内容包括：

- 系统名称
- 当前版本号
- 更新时间
- 本版本新增或修复内容

示例：

```text
系统名称：轻课管教培课时管理系统
版本号：V1.0
更新时间：2026-05
版本说明：
1. 支持新生报名、上课记录、续费、退费、流水管理
2. 支持本地自动保存、导出、导入、绑定数据文件
3. 支持班级、教师、收费模式等基础信息管理
```

### 2. 联系方式.txt

建议内容包括：

- 售后联系微信/QQ/手机号
- 工作时间
- 你提供的支持范围

示例：

```text
技术支持：________
联系方式：________
服务时间：工作日 9:00 - 18:00
说明：
1. 提供基础安装与使用指导
2. 提供系统Bug修复更新
3. 不包含个性化定制开发
```

---

## 五、建议交付前的检查项目

正式打包前，建议你检查以下内容：

1. 登录页是否正常
2. 左上角品牌图标和系统名称是否正确
3. 使用说明文案是否正确
4. 数据工具是否正常
5. 导出 / 导入是否正常
6. 当前版本是否已备份到 GitHub
7. 系统内是否还残留测试性文案
8. 默认演示数据是否已经清理

---

## 六、建议的交付方式

### 方式一：压缩包交付

适合本地版最常见的交付方式：

1. 将整个 `轻课管教培课时管理系统_V1.0` 文件夹打包
2. 发给客户
3. 客户解压后，打开 `系统程序 -> index.html`

### 方式二：网盘交付

可以将交付包上传到网盘，再发给客户下载。

建议同时告诉客户：

- 解压后不要随意更改文件夹层级
- 首次使用先阅读 `快速上手.txt`

---

## 七、建议的版本维护方式

后续如果继续更新版本，建议你保留：

- 当前正式版
- 上一个稳定版
- 当前开发版

简单理解就是：

- `可交付版本`
- `历史稳定版本`
- `正在修改的版本`

这样即使后面有新改动不满意，也能快速回到上一个稳定交付版。
